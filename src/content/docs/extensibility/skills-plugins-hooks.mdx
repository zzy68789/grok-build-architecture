> **结论：** Skills、Plugins 与 Hooks 位于不同扩展层：Skill 通过受控发现把说明和资源送入 Agent 能力面；Plugin 以包和市场元数据组织可安装扩展，并需要信任扫描；Hook 在运行时工具调用前后拦截、拒绝或旁路通知。三者不能用“都是插件”概括。

### Skill：按需注入的方法知识

Skill 由说明文件和可选资源构成，发现阶段先确定候选，再按规则读取。它适合编码流程、领域知识和工具使用约定。路径来源必须受控：符号链接、越界引用或不可信内容不能仅因文件名正确就获得信任。

### Plugin：可分发能力包

Plugin 解决安装、版本、来源和市场组织。一个插件可以携带多种能力，但“市场中可见”不等于“本地已安装”，安装也不等于默认信任。scanner 提供静态检查层，运行时仍应遵循权限和 sandbox。

### Hook：工具协议上的拦截点

pre-tool hook 可以在真实副作用发生前检查或拒绝，适合策略、审计和参数治理；通知型 hook 不应阻塞主调用，适合记录和外部提醒。hook 自身失败时，需要明确采用 fail-open 还是 fail-closed，不能让错误语义含糊。

**推断：** 三层拆分允许“知识怎样进入模型”“扩展怎样被分发”“动作怎样被治理”独立演进，也防止市场安装逻辑渗入每次 tool call 的热路径。

## 信任链

扩展内容可影响模型决策，hook 甚至能接触工具参数，因此来源、扫描和权限都必须可追溯。静态 scanner 能发现一部分危险结构，无法证明运行时行为安全；权限管理和 OS sandbox 仍是最终执行链的一部分。

## 边界与失败模式

- 同名 Skill 来自多个范围时必须有确定优先级，否则加载结果不可复现。
- 插件更新可能改变其权限需求，不能把旧版本信任无条件继承给任意新内容。
- 阻塞 hook 超时会拖住工具和 Agent loop；非阻塞 hook 则不能承担必须完成的安全裁决。
- hook 递归触发工具时可能形成循环，需要协议层限制或重入保护。

Skill discovery、marketplace/scanner 与 hook dispatcher 分别提供三层证据。它们最终汇入工具和 prompt 系统，但拥有不同生命周期与风险面。
