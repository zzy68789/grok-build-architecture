传输可以不同，但上层状态机相似：创建连接、协商协议、发现能力、维持运行状态、处理调用、关闭与重连。OAuth 是某些远端 server 的认证分支，不是所有 MCP 连接的固定步骤。

## 为什么需要命名空间

多个 server 可能都暴露 `search` 或 `read`。统一注册表要求名称可唯一解析，因此集成层用稳定分隔规则编码 server 和 tool。模型看到的是本地安全名称，调用时再反解并路由。

名称转换不是纯展示细节：它必须与注册、权限元数据和结果回传保持一致。若 server 名或工具名包含特殊字符，还要确保编码和解析可逆。

## 能力发现与信任

远端 schema 描述工具参数，却不能证明 server 行为可信。MCP 工具仍应进入本地权限体系；连接端点、认证 token 和工具结果都需要按外部输入处理。server 热重连后能力集合也可能变化，注册表需要更新而非永久缓存第一次发现。

**推断：** `McpState` 集中保存 server 和工具映射，是为了让异步重连、能力刷新与调用路由共享同一事实源，避免每次 tool call 重新扫描所有连接。

## 边界与失败模式

- transport 已连接但 initialize 或 list-tools 失败时，server 不能视作可用。
- OAuth token 过期可能只在调用阶段暴露，需要区分认证失败与工具业务错误。
- 远端工具 schema 变化会使已有模型上下文过期，当前 turn 可能仍持有旧 definition。
- server 关闭或超时应返回可理解错误，不能让整个 Agent loop 永久等待。

## 与官方文档的区别

官方扩展文档提供 MCP 配置和使用方式；本文不维护配置字段表，只解释配置进入哪个生命周期、外部工具怎样进入本地注册表。
