## 上下文组装顺序

  C[Session config / mode] --> B

项目规则的作用域通常与目录层级相关：越接近目标文件的规则越具体。系统需要知道当前工作目录、目标路径或会话 workspace，才能选择正确规则集合；只在仓库根目录搜索一次会丢失嵌套约束。

## 规则不是普通聊天历史

规则来源需要保留边界和来源信息，原因有三：

1. 更新工作目录时，适用规则可能变化。
2. 上下文压缩时，稳定指令不应被误当作可随意概括的旧对话。
3. 调试冲突时，需要知道某条要求来自系统、用户还是项目文件。

prompt builder 把这些来源汇合，但“汇合”不意味着优先级相同。上层协议和安全约束不能被较低层项目文本覆盖。

## AGENTS.md 的架构意义

`AGENTS.md` 把仓库知识放回代码附近：构建命令、目录约定、测试范围和编码规则可以随源码版本演进。对 Agent 来说，它是运行时上下文发现机制；对团队来说，它也是面向人和工具的可审计约定。

**推断：** 将兼容来源先 partition 再组装，说明实现希望保留多生态规则输入，同时避免把每种文件格式硬编码进采样层。

## 边界与失败模式

- 规则文件过长会消耗 token，并挤压任务本身所需的代码上下文。
- 目录移动后若沿用旧缓存，可能应用错误作用域的规则。
- 多来源冲突时，不应依靠文本出现顺序猜优先级；要查看 builder 的分层语义。
- 项目规则是指令输入，不是可信代码；其中的外部命令仍受权限与 sandbox 约束。

## 与官方文档的区别

官方材料适合查规则文件怎样写；本文聚焦发现、分区和组装过程。具体支持的文件名与配置开关可能随版本变化，应以官方说明和固定源码共同核验。
