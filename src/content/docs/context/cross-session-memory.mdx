## 从写入到召回

<Mermaid label="Memory 数据管线" chart={`flowchart LR

切分决定检索粒度。块太大，相关片段会被无关内容稀释；块太小，事实失去上下文。embedding 适合捕捉语义近似，关键词检索则能保留路径、符号、错误码等精确字面信号，混合排序正是为了平衡二者。

## 为什么 Memory 不直接等于 Prompt

持久化内容可能过期、互相冲突或来自旧项目状态。搜索结果需要经过相关度、数量、作用域和预算过滤，之后才作为有来源的上下文输入。当前仓库源码和用户明确指令通常比历史记忆更接近事实。

这也意味着“写入成功”和“以后一定被召回”不是同一保证。召回取决于查询、索引可用性、排序和预算。

## Dream 的角色

Dream 更像离线整理：在满足数量、时间或资源门槛时，合并重复信息、提炼长期知识或调整记忆结构。它与当前 turn 解耦，避免用户等待每次整理；但异步执行也要求对并发写入、失败重试与幂等保持谨慎。

**推断：** Dream 命名体现“从事件记录提炼知识”的意图，但具体产物仍应按代码数据结构理解，而不是类比成人类记忆机制。

## 边界与失败模式

- embedding 模型或版本变化会改变向量分布，旧索引可能需要重建。
- 记忆中的路径、API 和项目结论会随仓库演进而失效，召回时需核验当前源码。
- hybrid rank 可以改善召回，但不能证明事实真实，也不能替代指令优先级。
- Dream 后台失败不应阻断当前会话；同时也不能静默丢失未处理状态。
