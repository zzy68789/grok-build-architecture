> **结论：** Grok Build 采用多 crate Rust workspace，把界面、会话编排、模型采样、工具、权限/工作区与持久化拆开；端到端路径的中心不是 TUI，而是 `Leader → Session → Sampler/Tools` 的状态流。

- `xai-grok-pager-bin` 负责进程启动和运行模式分派。
- `xai-grok-pager` 负责 TUI 状态、`Action`、`Effect` 与事件循环。
- `xai-grok-shell` 是会话编排中心，包含 Leader、ACP session、prompt 构建、压缩和 subagent 协调。
- `xai-grok-sampler` 把模型端的流式响应变成运行时事件。
- `xai-grok-tools` 提供注册表、桥接层和内建工具实现。
- workspace、sandbox、memory、compaction 等 crate 把高风险或高复用能力隔离出来。

这里的“shell”不是操作系统 shell 的同义词，它更接近 Agent runtime。命令执行只是工具系统中的一种动作。

数据跨层时会改变形态：键盘事件先成为 UI `Action`，客户端请求再成为 session 输入，模型输出成为 sampler event，工具调用最终成为结构化 result。这些转换点也是排障时最有价值的观测点。

## 设计边界

`Leader` 的单实例设计让多个客户端共享 Agent 状态，但也使 Leader 生命周期成为连接与恢复的关键。工具注册表将“模型知道哪些工具”与“具体工具怎样执行”解耦；权限和 sandbox 则把策略判断与 OS 级执行限制分开。

**推断：** workspace 的细粒度拆分不仅服务编译复用，也是在约束依赖方向。例如压缩引擎与 token 估算可以独立测试，避免把全部会话运行时拉入算法层。

## 边界与失败模式

- crate 边界不代表运行时进程边界，多数模块仍处于同一进程和异步运行时中。
- UI 收到的事件可能晚于 session 内部状态变更；不能只凭一帧界面推断底层状态。
- 工具定义成功进入模型上下文，不代表调用一定通过权限或参数校验。
- Leader 重连、session 持久化和 checkpoint 分别解决不同层次的恢复，不应混为一种“保存”。
