> **结论：** Grok Build 不是“在终端里包一层聊天界面”，而是一个以终端为主要交互面、能持续观察上下文并执行工程动作的 agentic coding system。它把模型采样、工具执行、权限决策、工作区状态与会话恢复组合成同一条运行链。

## 源码入口

仓库 README 直接给出产品定位，并把交互入口指向终端；同一文件还说明主要实现位于 Rust workspace。这个定义很重要：UI 只是入口，系统的核心价值在于受控执行与状态闭环。

真正的可执行入口位于 pager binary。`main` 在完成平台初始化、日志与错误处理之后，将工作交给异步启动流程；这说明 CLI、TUI、服务模式共享的是同一套进程级基础设施，而不是互不相关的程序。

## 一个系统，而不是单轮问答

<Mermaid label="Grok Build 的能力闭环" chart={`flowchart TD
  I[自然语言目标] --> C[收集项目上下文]
  C --> S[模型采样]
  S --> D{需要动作?}
  D -->|否| O[输出答案]
  D -->|是| G[选择并调用工具]
  G --> P[权限与沙箱]
  P --> E[读取 / 修改 / 执行]
  E --> H[结果回到会话历史]
  H --> S`} />

与普通聊天程序相比，这条链路多出三个结构性条件：

1. **动作必须可描述。** 文件、命令、搜索、任务等能力以工具协议进入模型可见上下文。
2. **动作必须可裁决。** 工具不是拿到参数就直接执行，还要经过权限策略与操作系统隔离能力。
3. **结果必须可续接。** 工具结果、流式文本、checkpoint 与持久化状态要重新汇入当前 `Session`，供下一次采样使用。

因此“Agent”不是某个孤立类名，而是一条由多个 crate 协作完成的运行性质。

## 关键实现含义

终端优先并不等于只支持 TUI。源码同时提供 headless、stdio 与 serve 等运行方式；它们改变输入输出适配层，却尽量复用 `Leader`、`Session`、Sampler 和工具系统。这种分层让同一 agent 状态可以服务不同客户端。

**推断：** 将产品描述为 terminal-native 而非 terminal-only，更符合源码结构。终端是默认体验与设计中心，但核心运行时已经抽离出可被协议或服务入口驱动的边界。

## 边界与失败模式

- 模型生成内容不等于操作已经完成；只有工具结果返回后，外部状态才有可验证变化。
- 自动模式不会消除所有安全边界，OS sandbox 和工具自身校验仍可能拒绝动作。
- 固定快照只能解释该提交的实现；官方服务端模型行为、未公开基础设施不在本站证明范围内。
- TUI 显示异常可能来自渲染层，也可能来自更上游的流式事件或会话状态，排查时应沿调用链定位。
